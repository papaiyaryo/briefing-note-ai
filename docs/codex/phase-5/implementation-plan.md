# Phase 5 実装計画: MVP 品質改善・テスト

## Phase 目的

Phase 4 で完成した「アップロード → ダミー OCR → Markdown 生成 → 編集 → `.md` ダウンロード」までの
ローカル完結フローに対して、テストと自己レビューを行い、デモ・ポートフォリオとして見せられる品質に近づける。

この Phase では新機能を追加しない。既存ロジックの **品質保証(テスト)**、**安全性確認(セキュリティ/プライバシー)**、
**ドキュメント整備(README・スクリーンショット)** の 3 種類の作業のみを行う。

## 対象 Issue 一覧

| Issue | タイトル | 優先度 | 役割 |
| --- | --- | --- | --- |
| #27 | Markdown 生成ロジックの単体テストを追加する | High | テスト(ロジック) |
| #28 | UI コンポーネントの基本テストを追加する | Medium | テスト(UI) |
| #29 | E2E テスト方針を整理する | Medium | テスト方針(ドキュメントのみ) |
| #30 | セキュリティ・プライバシー確認を実施する | High | 安全性確認 + ドキュメント |
| #31 | MVP セルフレビューと README 更新を行う | High | セルフレビュー + README |
| #32 | MVP デモ用スクリーンショットを追加する | Low | ポートフォリオ用素材 |

## 実装順序と依存関係

```text
#27 (Markdown 生成ロジックの単体テスト)
#28 (UI コンポーネントの基本テスト)
  └─> いずれも Phase 4 (#22-#26) に依存。テスト対象の関数 / コンポーネントが既に存在する。
#29 (E2E テスト方針整理。実装は行わない)
#30 (セキュリティ・プライバシー確認)
  └─> Phase 4 MVP フロー全体(ログ出力・.env 取り扱い・データフロー)に依存。
        README に「外部送信なし」の前提を明記する作業を含むため、#31 より先に完了させる。
#31 (MVP セルフレビューと README 更新)
  └─> #27, #28, #29, #30 の結果(テストが揃っている、安全性が確認済み)を前提に
        README を MVP 利用手順向けに整理する。#30 で追記した安全性に関する記述は
        #31 で重複させず、必要なら参照するだけにする。
#32 (デモ用スクリーンショット)
  └─> UI が変わらない前提で独立して進められるが、README へのリンクは #31 の後に追加すると
        README の構成変更との競合が起きにくい。
```

推奨実装順: **#27 → #28 → #29 → #30 → #31 → #32**

理由:

- #27 / #28 はテスト追加のみで他 Issue に影響しないため最初に進められる。
- #29 はドキュメントのみで実装依存がなく、テストの後に方針として整理すると一貫性がある。
- #30 はログ出力や `.env` の扱いなど Phase 4 までの実装結果を確認する作業であり、
  README に追記する「外部送信なし」の文言は #31 の README 全体更新の土台になる。
- #31 は #27-#30 の結果(テスト状況・安全性確認結果)を踏まえて README を仕上げる総括的な Issue。
- #32 はスクリーンショットの追加のみで、README 構成が固まった #31 の後に行うと差分が安定する。

各 Issue は **1 Issue = 1 Branch = 1 PR**。複数 Issue を 1 PR にまとめない。

## 共有設計判断(Phase 5 全体で固定)

### 1. 新機能は追加しない

Phase 5 はテスト・安全性確認・ドキュメントのみを扱う。Phase 4 までに実装したロジックや UI の挙動を変更しない
(明確なバグでない限り、テストのために実装を変更しない)。

### 2. テスト方針

- 単体テスト(#27, #28)は既存の Vitest 構成(`tests/**/*.test.ts`)に追加する。新しいテストランナーは導入しない。
- UI コンポーネントテストはレンダリングと主要な操作(ボタン押下・入力)の確認に限定し、
  ビジュアルリグレッションや外部 API を使うテストは行わない。
- E2E(#29)は方針整理のみで、Playwright 等の実装・導入は Phase 5 では行わない。

### 3. セキュリティ・プライバシー方針(#30 が中心)

- `docs/architecture.md` の「セキュリティとプライバシー」節と `AGENTS.md` の「API / Secret Rules」を正とする。
- 確認観点は 3 つに限定する: (1) ログ出力に画像・OCR全文・Markdown・個人情報を含まないか、
  (2) `.env` や API キーが repo に含まれていないか、(3) MVP が外部送信を行わない前提が
  README / docs に明記されているか。
- 本番運用を想定した監査体制や認証・権限管理は Phase 5 のスコープ外とする(Issue #30 の Out of Scope と一致)。
- Phase 6(OpenAI 連携)着手前に同じ観点で再確認する(Issue #30 の Notes と一致)。

### 4. README 更新の分担

- #30: 「外部送信なし」という MVP の前提と、画像・OCR・Markdown の保存方針(ブラウザ内のみ、リロードで消える等)を
  README に追記する。新しいセクションを 1 つ追加する程度の小さな変更にする。
- #31: README 全体を MVP の利用手順・機能一覧・Out of Scope・今後の予定で整理し直す。
  #30 で追記した内容と重複させず、必要な場合は見出しの位置を調整する。

### 5. ドキュメントのみの Issue の扱い

#29 と #30 の一部(README 追記を除く)はコードを変更しない。テストやビルドに影響しないため、
検証は `npm run lint` / `npm run format:check` 程度で十分とする。

## Phase 全体の検証方針

各 Issue の PR で、変更内容に応じて以下を実行する。

```bash
npm run lint
npm run typecheck
npm test
npm run build
```

- #27, #28: 上記すべてに加え、追加したテストが green であることを確認する。
- #29: ドキュメントのみのため `npm run lint` / `npm run format:check` で十分(コード変更がない場合)。
- #30: コード変更がない場合は #29 と同様。ログ出力確認は `grep` 等での手動確認結果を PR に記録する。
- #31: README 変更後、リンク切れや手順の矛盾がないか目視確認する。
- #32: 画像追加後、README からの参照が表示されることを確認する。

## スコープ外(Phase 5 では実装しない)

- 本番監査体制、認証・権限管理(Issue #30 の Out of Scope)
- E2E テストの実装そのもの(Issue #29 の Out of Scope、方針整理のみ)
- LLM 出力のテスト、外部 API を使うテスト(#27, #28, #29 の Out of Scope)
- ポートフォリオ向けの詳細な仕上げ、デモ動画 / GIF(#31, #32 の Out of Scope、Phase 10 で対応)
- Post-MVP 機能の実装・紹介

## リスク / 留意点

- **テスト追加による実装変更の誘発**: #27/#28 でテストしやすくするために実装を不必要に変更しない。
  テストできない箇せは設計判断として記録し、実装変更は別 Issue で検討する。
- **README の二重編集**: #30 と #31 がどちらも README を編集するため、実装順(#30 → #31)を守り、
  #31 の PR 作成前に #30 がマージされていることを確認する。順序が守れない場合はコンフリクトに注意する。
- **ログ出力確認の再現性**: 現時点(Phase 4 終了時点)では `console.*` 呼び出しは存在しないため #30 の
  該当 Acceptance Criteria は実質的に満たされているが、Phase 6 以降(API 連携)で再発しないよう、
  確認手順をドキュメント化しておく。
- **「外部送信なし」の前提が将来変わる**: Phase 6 (OpenAI) / Phase 7 (Google Drive) で前提が変わるため、
  README / docs には「現時点(MVP)では」という条件を明記し、将来更新が必要なことが分かるようにする。
