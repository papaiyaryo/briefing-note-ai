## Issue 概要

画像・OCR テキスト・生成 Markdown の扱いが就活情報として安全かを確認する。
本 Issue は基本的に**確認作業**であり、大規模なコード変更は想定しない
(`AGENTS.md` の API/Secret Rules、`docs/architecture.md` のセキュリティ節を基準にする)。

## スコープ

- ログ出力に画像内容や OCR 全文が含まれないか確認する。
- API キーや `.env` がコミットされていないか確認する。
- 外部送信なしの MVP 前提を README / docs に明記する。

## スコープ外

- 本番監査体制。
- 認証・権限管理。

## 共有設計判断(Phase 5 計画より)

- 確認の結果、ルール違反が見つかった場合は**本 Issue の PR 内で最小限の修正**を行う。
- 大きな修正が必要と判明した場合は別 Issue 化を提案し、Phase 5 を肥大化させない。
- 「外部送信なし」の明記は README 本文を中心に行い、詳細な記述は `docs/architecture.md` の
  既存セキュリティ節(`## セキュリティとプライバシー`)を参照する形にする(重複記述を避ける)。

## 実装ステップ

1. **`console.log` 等の出力確認**: `src/` 配下を `console.` で検索し、画像内容(Blob / dataURL)、
   OCR 全文、生成 Markdown 全文、ユーザー入力(企業名等)を不要に出力していないか確認する。
   - 確認対象の主なファイル: `src/components/BriefingNoteFlow.tsx`、`src/lib/dummyOcr.ts`、
     `src/lib/markdown.ts`、`src/lib/download.ts`。
   - 現状把握: これらのファイルはログ出力を行っていない見込みだが、確認は実施する。
2. **`.env` / API キーのコミット確認**:
   - `.gitignore` に `.env` が含まれているか確認する。
   - `.env.example` に実際のキー値(ダミーでないもの)が入っていないか確認する。
   - リポジトリ全体を `git log` ベースで検索し、過去のコミットに `.env` や鍵らしき文字列が
     含まれていないか確認する(含まれていた場合は Issue のスコープを超えるため、対応方針を
     ユーザーに相談する)。
3. **保存方針の明記確認**:
   - `docs/architecture.md` に「将来保存機能を追加するまでは、永続保存しない設計を優先する」旨が
     すでに記載されていることを確認する(現状記載あり)。
   - README にも MVP は外部送信・サーバー保存を行わない旨を一文で明記する
     (#31 の README 更新と合わせて反映してよいが、本 Issue でも確認観点として残す)。
4. 確認結果(問題なし、または対応した修正点)を本 Issue の PR 本文にチェックリストとして記録する。
