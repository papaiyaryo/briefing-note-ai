# E2E Test Policy

## 目的

このドキュメントは、Phase 5 Issue #29 の範囲として、MVP のアップロードから `.md` ダウンロードまでを将来 E2E で確認するための方針を定義します。

Issue #29 では方針整理のみを扱い、E2E テストコード、Playwright 設定、CI workflow への E2E 組み込みは実装しません。

## 対象にする MVP フロー

E2E では、`docs/user-flow.md` の MVP 基本フローに沿って、まず正常系の一本道を対象にします。

1. トップ画面を開く
2. サンプル画像をアップロードする
3. アップロード画像のプレビューが表示されることを確認する
4. OCR を実行する
5. OCR 結果が表示されることを確認する
6. OCR 結果をもとに Markdown を生成する
7. 生成された Markdown を編集する
8. Markdown プレビューを確認する
9. 編集済み Markdown を `.md` としてダウンロードする

### E2E で確認する主な観点

- ユーザーが画面の案内に従って、アップロードからダウンロードまで迷わず到達できること
- OCR 結果表示、Markdown 編集、Markdown プレビュー、`.md` ダウンロードの各状態が画面上で確認できること
- 生成 Markdown が、少なくとも次の情報を混在させずに扱える UI になっていること
  - 紙のメモ由来の事実
  - 個人の感想
  - HR・社員が強調した点
  - 後で質問したいこと
  - ES・面接に使える観点
  - Web 由来の補足情報（MVP では取得しないため、空または未使用であること）
- ダウンロードされる `.md` ファイル名と内容が、MVP の仕様に沿っていること

## 初期導入時点では対象外にするフロー

Issue #29 および初期 E2E 導入時点では、次の項目は対象外にします。

- 画像アップロード失敗、OCR 失敗、Markdown 生成失敗などのエラー系網羅
- 複数画像アップロードや複数企業メモの連続処理
- Google Drive 保存
- Web 補足情報取得
- 企業比較
- 面接前復習モード
- ユーザーアカウント、認証、マルチデバイス同期
- 実 OpenAI API、Google Drive API、外部 OCR API を使うテスト

エラー系や Post-MVP 機能の E2E は、それぞれの機能 Issue で仕様と安全性を確認してから追加します。

## ツール候補

| 候補 | 長所 | 懸念 | 方針 |
| --- | --- | --- | --- |
| Playwright | Next.js と組み合わせやすい。Chromium / Firefox / WebKit を扱える。ネットワークモック、ダウンロード検証、スクリーンショット、トレースが使いやすい。GitHub Actions との連携例が多い。 | 初期設定とブラウザ依存パッケージの導入が必要。 | 第一候補にする。 |
| Cypress | UI 操作のデバッグ体験がよい。学習しやすい。 | マルチブラウザや複数タブ・ダウンロード検証の柔軟性では Playwright より制約を受ける可能性がある。 | 代替候補に留める。 |
| Puppeteer | Chromium 操作が軽量。Node.js から直接扱いやすい。 | E2E テストランナー、複数ブラウザ、レポート、CI 連携を個別に整える必要がある。 | 採用優先度は低い。 |

MVP の E2E では、ダウンロード検証と外部 API スタブ化を重視するため、現時点の採用候補は Playwright とします。

## 外部 API を使わない実行方針

E2E は秘密情報や外部サービスの状態に依存しない deterministic なテストとして実行します。

- Phase 4 の MVP フローでは、実 OCR / 実 LLM の代わりにリポジトリ内のダミーデータまたはサンプルデータを使う
- Phase 6 以降に OpenAI API Routes や OCR API Routes が追加された後も、E2E では Playwright のネットワークインターセプトでレスポンスをスタブ化する
- テスト入力は `docs/sample-data.md` など、リポジトリにコミット可能なサンプルだけを使う
- 実在企業の非公開メモ、個人情報、アップロードされた実画像は使わない
- CI artifact やログに、アップロード画像、OCR 全文、生成 Markdown、個人情報を保存しない

## 秘密情報を使わない方針

E2E の初期導入では GitHub Secrets を使いません。

- OpenAI API キー、Google OAuth クライアントシークレット、アクセストークンを不要にする
- `.env` は使わず、必要な設定がある場合も `.env.example` にダミー値または空値のみを記録する
- CI で外部 API を呼ばないため、課金、レート制限、外部障害の影響を受けない
- テストログには秘密情報、個人情報、実メモ本文を出力しない

将来 GitHub Secrets が必要になる E2E を追加する場合は、別 Issue で用途、保存する値、ログに出ないこと、代替スタブ可否を確認してから導入します。

## 導入タイミング

Issue #29 では E2E 方針の文書化までを完了条件にします。

E2E テスト実装は、Phase 5 の MVP 品質改善が完了し、Phase 4 の MVP フローが安定した後に、フォローアップ Issue として実施します。特に Phase 6 の OpenAI API 連携に入る前に、外部 API なしで MVP 正常系を確認できる E2E を追加することを推奨します。

導入時の最小作業は次の想定です。

1. Playwright と必要なブラウザ依存を devDependencies に追加する
2. `npm run test:e2e` などの npm script を追加する
3. サンプル画像またはテスト用 fixture を追加する
4. アップロードから `.md` ダウンロードまでの正常系 1 本を実装する
5. CI に E2E を追加するか、当初は手動実行に留めるかを Issue 内で決める

## CI への組み込み方針

初期導入時は、既存 CI の `lint` / `typecheck` / `test` / `build` を壊さないことを優先します。

- 最初は通常の単体テストとは分けて `npm run test:e2e` として実行する
- CI へ追加する場合は、通常チェックの後に独立した job として実行する
- 外部 API、GitHub Secrets、デプロイ済み環境に依存しない
- 失敗時にスクリーンショットやトレースを保存する場合も、サンプルデータだけが含まれることを確認する

## Acceptance Criteria との対応

- E2E の対象フローが docs に記録されている
  - 「対象にする MVP フロー」でアップロードから `.md` ダウンロードまでの正常系を定義しました。
- 導入タイミングが明確である
  - 「導入タイミング」で Issue #29 は方針整理まで、実装は Phase 5 完了後・Phase 6 着手前のフォローアップ Issue と定義しました。
- 秘密情報を使わないテスト方針になっている
  - 「外部 API を使わない実行方針」と「秘密情報を使わない方針」で、GitHub Secrets と実 API に依存しない方針を定義しました。
