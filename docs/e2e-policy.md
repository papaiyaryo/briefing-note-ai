# E2E テスト方針

## 目的

このドキュメントは、MVP のアップロードから `.md` ダウンロードまでの一連の操作を
将来 E2E (End-to-End) テストで確認するための方針を整理する。

Issue #29 (Phase 5) のスコープは方針の整理のみであり、E2E テストの実装は行わない。
実装は別 Issue として、後述の「導入タイミング」に従って起票する。

## 対象とする MVP フロー

`docs/user-flow.md` の基本フローのうち、E2E で確認する価値が高い「正常系の主要 1 本道」を対象とする。

1. トップ画面を開く
2. 企業説明会メモ画像をアップロードする
3. アップロード画像のプレビューを確認する
4. OCR を実行する(Phase 4 時点ではダミー OCR)
5. OCR 結果を確認する
6. Markdown を生成する
7. Markdown を編集する
8. 編集済み Markdown をプレビューする
9. 編集済み Markdown を `.md` としてダウンロードする

### 対象に含めるもの

- 上記 1 本道(happy path)が画面操作だけで最後まで完了すること
- ダウンロードされたファイル名・拡張子(`.md`)が期待通りであること
- 各ステップで主要な要素(プレビュー、OCR 結果、エディタ、ダウンロードボタン)が表示されること

### 対象外(MVP の E2E では扱わない)

- アップロード失敗・OCR 失敗・生成失敗などのエラー系の網羅(ユニット/コンポーネントテストで担保する。`05-01`, `05-02` 参照)
- Google Drive 保存、Web 補足、企業比較、面接前復習モードなど Post-MVP 機能のフロー
- 実 OpenAI / 実 Google Drive など外部サービスに依存するフロー
- ビジュアルリグレッション(スクリーンショット比較)

## ツール候補比較

| 観点 | Playwright | Cypress | Puppeteer |
| --- | --- | --- | --- |
| Next.js (App Router) との実績 | 実績が多く、公式サンプルやコミュニティ事例が豊富 | 利用可能だが SSR/Server Actions 周りで追加設定が必要な場合がある | ブラウザ自動化ライブラリであり、テストランナーやアサーションは別途必要 |
| マルチブラウザ対応 | Chromium / Firefox / WebKit を標準サポート | 主に Chromium 系、WebKit は実験的 | 標準では Chromium のみ(Firefox は実験的) |
| ネットワークモック | `page.route()` で柔軟にリクエストを横取り・スタブ可能 | `cy.intercept()` で同様に可能 | 可能だが低レベル API を自前で組み立てる必要がある |
| 並列実行 / CI 連携 | 組み込みの sharding、公式 GitHub Action あり | 並列実行は Cypress Cloud(有料)前提が強い | 自前でランナー・並列化を構築する必要がある |
| TypeScript 親和性 | 公式型定義があり高い | 高い | 高い(ただしテスト構成は別途必要) |
| セットアップコスト | `npm init playwright` で一式生成、Vitest と併存可能 | 同程度に簡単 | テストランナー(Vitest 等)とアサーションライブラリの組み合わせが必要 |
| 学習コスト | 中 | 低〜中 | 高(テスティングフレームワークではないため) |

### 選定

**Playwright を採用候補とする。**

理由:

- テストランナー・アサーション・ネットワークモック・マルチブラウザ・並列実行(sharding)が一式に含まれ、追加の有料サービスを必要としない。
- 公式 GitHub Action があり、`docs/ci-policy.md` が前提とする GitHub Actions 上の CI に組み込みやすい。
- `page.route()` によるリクエストの横取りが容易で、後述の「外部 API なしで実行する方針」を満たしやすい。
- 既存スタック(Next.js / TypeScript / Vitest)と競合せず、Vitest はユニット/コンポーネントテスト、Playwright は E2E という役割分担にできる。

Cypress は有力な代替候補だが、無料枠での並列実行や WebKit 対応が弱く、現状では採用見送りとする。
Puppeteer はテストランナーを持たないため、最小構成での導入コストが他候補より高いと判断する。

## 外部 API なしで実行する方針

- Phase 4 時点の MVP フローは、ダミー OCR (`src/lib/dummyOcr.ts` の `runDummyOcr`) と
  ローカル完結の Markdown 生成ロジックのみで完結しており、ネットワークアクセスを行わない。
  E2E はこの deterministic な(同じ入力 → 同じ出力の)状態に対して実行する。
- Phase 6 以降で実 OpenAI API 連携(サーバー側 API Route)が入った後も、E2E テストは
  実際の外部 API を呼び出さない。Playwright の `page.route()` で API Route へのリクエストを
  インターセプトし、固定のレスポンス(サンプルデータ)を返すスタブに置き換える。
- Google Drive 保存などの Post-MVP 機能についても、E2E から実 API を呼び出すことはしない。
  対象に含める場合は同様にネットワーク層でスタブ化する。
- これにより、E2E テストは CI 環境やローカル環境を問わず、ネットワーク接続や外部サービスの
  可用性に依存せず安定して実行できる。

## 秘密情報を使わないテスト方針

- E2E テストは API キー・OAuth クライアントシークレット・アクセストークンなどの秘密情報を
  一切必要としない。テスト対象がダミー OCR / スタブ化した API Route のみであるため。
- GitHub Secrets を E2E 用に追加することはしない(`docs/ci-policy.md` の「秘密情報とログ」方針を継承する)。
- テストで使用する入力データ(画像・企業名・OCR テキストなど)は、実在企業や個人情報を含まない
  サンプルデータ(`docs/sample-data.md` 想定のもの)のみを使用する。
- アップロードする画像も、リポジトリに含む架空のサンプル画像、または実行時にテストコード内で
  生成するダミー画像とし、実際のユーザーデータを使わない。
- 生成された Markdown やテスト実行時のログに、秘密情報や個人情報が含まれないことをコードレビューで確認する。

## 導入タイミング

E2E テストの**実装**は本 Issue (#29) のスコープ外とし、次の方針で別 Issue として導入する。

- **タイミング: Phase 5 完了後、Phase 6 (OpenAI API 連携) の実装開始前。**
- 理由:
  - Phase 5 終了時点で MVP の正常系フロー(アップロード〜ダウンロード)が確定し、
    ダミー OCR によって deterministic に再現できるため、E2E の対象が安定する。
  - Phase 6 で実 API 境界(サーバー側 API Route)が追加されると、フロントエンドと
    バックエンドの結合点が増え、回帰のリスクが上がる。E2E の happy path をその前に
    固定しておくことで、Phase 6 以降の変更がコア UX を壊していないかを検証しやすくする。
  - Phase 6 の API Route 追加後は、E2E のネットワークスタブ対象(`/api/ocr` 等)を
    追加・更新するだけで済むようにしておく。
- 導入時の Issue 化:
  - Phase 6 着手前に、新規 Issue(例: 「Playwright による MVP E2E スモークテストを実装する」)を
    Phase 5 のフォローアップとして起票する。
  - `AGENTS.md` の `1 Issue = 1 Branch = 1 PR` に従い、E2E 実装はこの新規 Issue 内で行う。
  - 実装後の CI 組み込みは `docs/ci-policy.md` の「実行するチェック」に E2E ステップを
    追加する形で反映する(npm scripts へ `test:e2e` 相当を追加するなど)。
